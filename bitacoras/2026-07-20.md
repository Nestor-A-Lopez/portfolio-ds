## Qué entendí bien

- `git status` nos revela estado de los archivos dentro de los repositorios, lo que incluye: si los archivos están siendo o no monitoreados y, de estar monitoreados, si han sido o no modificados o si se están listos para guardar sus cambios de manera permanente (_staged_)
- Las ramas son formas de nombrar _commits_. De este modo el usuario sabe con qué versión del proyecto está trabajando. Con cada rama se crea un puntero que señala al _commit_ más reciente de esa línea de trabajo
- Entre los comandos básicos para trabajar con ramas se encuentran:
  - `git branch <nombre-de-la-raman>`: para crear una nueva rama
  - `git switch <nombre-de-la-rama>`: para moverse entre ramas
  - `git branch -m <antiguo-nombre> <nuevo-nombre>`: para renombrar una rama
  - `git branch -d <nombre-de-la-rama>`: para eliminar una rama
- El puntero `HEAD` es un puntero que apunta al puntero de una rama y es una forma de saber en qué rama se encuentra el usuario; moverse entre ramas significa cambiar a qué puntero apunta `HEAD`
- Para fusionar ramas se utiliza el comando `git merge`. Existen dos tipos de fusiones: _fast-forward merge_ y el _three way merge_. El _fast-forward merge_ sucede cuando las líneas de trabajo de las ramas no se desarrollaron en paralelo, de modo que todo el historial de _commits_ de una se puede encontrar la otra, como en una secuencia lineal. De modo que ejecutar `git merge` solo hace que el puntero de la rama avance hasta apuntar al _commit_ más reciente. El _three way merge_ sucede cuando dos ramas se desarrollaron en paralelo y no comparten el mismo historial de _commits_, al _commit_ en donde dos ramas se bifurcaron se le conoce como _merge base_ y se puede consultar con `git merge-base <rama1> <rama2>`. Ya que en el _three way merge_ una línea se desarrolló independientemente de la otra y viceversa, pueden existir cambios entre versiones que causen conflictos al tratarse de fusionarlas. Un conflicto no es un error, sino una situación en el que el algoritmo de Git no puede distinguir qué cambios mantener y qué cambios eliminar. Por ello, de darse el caso, Git nos pedirá decidir qué cambios se desean guardar. En el _three way merge_, ya que existen dos líneas de trabajo, cada una con un historial de _commits_ diferente, es necesario la generación de nuevo _commit_: el _commit merge_. Este se diferencia de los demás _commits_ por tener dos _commits_ padres —uno por cada línea de trabajo— en lugar de solo uno. Así el historial de cambios de ambas líneas de trabajo se guardan en uno solo _commit_
- Entre las buenas prácticas con respecto a las ramas, se recomienda
  - No trabajar sobre la rama principal
  - Llevar los cambios a la rama principal constantemente
  - No reutilizar ramas
  - En lugar de tener una rama que abarque muchas características, es mejor tener muchas ramas que resuelvan pequeñas características
  - Eliminar ramas que ya no estén en uso
  - Crea nuevas ramas a partir de la rama principal
- `git log` nos muestra el historial de _commits_ completo, incluyendo el _hash_, autor y mensaje de cada uno de ellos. Existen diferentes banderas para modificar la cantidad de información y la forma en que esta se presenta al usuario. Entre los más importantes se encuentran:
  - `--oneline`: nos muestra cada commit en el historial de cambios en una sola línea
  - `--graph`: añade una gráfica con líneas de texto para poder visualizar de mejor forma las ramas del proyecto
  - `--all`: permite ver todas las ramas del proyecto y no solo la rama desde la que se ejecutó el comando (y a donde apunta _HEAD_)
- El estado _detached HEAD_ permite que el puntero `HEAD` apunte al _hash_ de un _commit_ anterior de esa línea de trabajo. Lo que permite explorar un estado anterior de los archivos del proyecto
