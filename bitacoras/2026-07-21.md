## Qué estudié

- ¿Cómo operar los archivos dentro del repositorio (restaurarlos, renombrarlos, o eliminarlos)?
- ¿Cómo operar con los _commits_ (cambiar el mensaje, deshacerlos)?
- ¿Cómo usar a _HEAD_ como referencia relativa?

## Qué entendí bien

- Para investigar diferencias entre diferentes pares de objetos se utiliza el comando `git diff`. Por defecto, `git diff` compara el estado de todos los archivos en el proyecto en el directorio de trabajo y el guardado en el _index_. Sin embargo, según sea lo que se desea comparar y entre qué áreas de trabajo, se introducen diferentes argumentos y banderas:
  - `git diff <archivo1>`: muestra las diferencias entre el estado de solo un archivo en el directorio de trabajo y su versión en el _index_
  - `git diff <rama1> <rama2>`: muestra las diferencias entre los _commits_ más recientes de sus respectivas ramas
  - `git diff <hash-commit1> <hash-commit2>`: muestra las diferencias entre dos diferentes _commits_ del estado del proyecto archivos
  - `--staged`: compara el estado del proyecto en el _index_ y el _repositorio_
  - `--word-diff`: sin esta bandera, Git muestra todas las líneas en donde haya encontrado diferencias, aún si la diferencia fue solo una palabra. Con ella, Git solo resalta las palabras que cambiaron en cada línea
- Es posible hacer referencia a un _commit_ con ayuda de _HEAD_, sirviendo como un apuntador relativo. Con `HEAD~n` se está haciendo referencia al _hash_ del antecesor número $n$ del _commit_ al que apunta actualmente _HEAD_: `HEAD~1` apunta al _commit_ padre, `HEAD~2` apunta al _commit_ abuelo y así sucesivamente. En el caso de un _merge commit_ (que tiene dos padres), se utiliza `HEAD^1` para referenciar al primer padre (el de la rama principal) y `HEAD^2` para referenciar al segundo padre (el de la _feature branch_)
- El comando `git restore` funciona como una forma de revertir los cambios en el contenido de un archivo. Por defecto, `git restore <archivo>` deshace los cambios realizados en el directorio de trabajo con su versión almacenada en el _index_. Por ello, este comando funciona únicamente para archivos que están siendo rastreados. Al igual que con `git diff`, según sean los argumentos o banderas, actuara de diferente forma:
  - `--staged`: si después de hacer `git add`, pero antes de hacer `git commit`, te arrepientes de haber añadido los cambios al _index_, esta bandera hace que esos cambios añadidos se eliminen al hacer que el _index_ desvincule el archivo con el _blob_ al que estaba asociado y lo vuelve a vincular al _blob_ al que está asociado en el último _commit_
  - `--source=<commit>`: en escencia, reescribe el contenido de un archivo en el directorio del trabajo con el contenido al que está asociado el archivo en el _commit_ dado como fuente
- `git mv <nombre1> <nombre2>`: permite que el _index_ se actualice con el nuevo nombre de un archivo al este ser renombrado por el sistema. Este comando realiza dos en cadena: `mv <nombre1> <nombre2>` y `git add <nombre2>`. De no hacerse, Git interpreta la acción como si `<nombre1>` hubiera sido eliminado y `<nombre2>` hubiera sido agregado al proyecto.
- `git rm <archivo>`: permite a Git saber que un archivo ha sido eliminado. Este comando realiza dos en cadena: `rm <archivo>` y `git add <archivo>`. Esto es necesario ya que de no hacerse, el _index_ no se entera de esta acción y sigue tratando de rastrear un archivo que ya no existe en el directorio de trabajo.
- Para añadir cambios que fueron olvidados en el último _commit_, se usa `git commit --amend`. Este comando no reescribe el _commit_, sino que crea uno nuevo con el contenido actual del _index_. De hecho, si solo se desea cambiar el mensaje con el que se declaró un _commit_, se suele usar `git commit --amend -m "nuevo-mensaje"`. El _commit_ remplazado al no estar vinculado con ninguna línea de trabajo luego puede ser tomado por basura por Git y ser eliminado.
- Para deshacer los cambios por _commits_ existentes, se pueden usar dos comandos: `git reset` o `git revert`.
  - `git reset <commit>`: revierte los cambios hechos en el proyecto hasta el _commit_ referenciado. Todos los cambios hasta entonces son almacenados en diferentes lugares según sea la bandera dada:
    - `--soft`: guarda los cambios en el _index_, así que estos pueden ser declarados nuevamente en un solo _commit_
    - `--mixed`: este es el comportamiento por defecto del comando si no se le entrega ninguna bandera. Guarda los cambios en el directorio de trabajo, así estos pueden volver a ser stageados con `git add`
    - `--hard`: no guarda los cambios en ningún lugar, por lo que todos los cambios que se dieron desde el _commit_ que fue dado como referencia se perderán
  - `git revert <commit>`: revierte los cambios hechos en el proyecto por un _commit_ en específico, pero para esto genera un nuevo _commit_ que deshace los cambios. Así, deshacer estos cambios queda dentro del historial de cambios, lo que lo hace una acción mucho más segura que `git reset`
